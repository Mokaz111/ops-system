# 云原生可观测性监控平台 — 总体设计文档（v3 现网对齐版）

## 1. 文档目的

描述 `ops-system` 当前已实现的控制面能力、模块边界、数据/控制流与非功能性约束，作为现网运行与后续迭代的基线。

约束：

- 文档与代码冲突时，以代码为准，并修正文档
- 不描述任何已废弃的阶段性方案；历史阶段计划见 `99-TCOP对齐改造实施计划.md`（已完结归档）

## 2. 系统定位

`ops-system` 是云原生可观测性平台的控制面，对齐腾讯云 TCOP 形态，统一管理：

- 监控（VictoriaMetrics 全栈：VMCluster / VMAgent / VMRule / VMPodScrape / VMServiceScrape）
- 日志（VictoriaLogs）
- 可视化（Grafana 多主机）
- 接入模板（采集 + 告警 + 大盘三合一模板市场）
- 指标库（指标字典 + 模板关联）

**告警引擎不自研**：告警计算/通知投递由 N9E 独立运行，平台只做集成接口与跳转入口。

## 3. 总体架构

```
                  ┌─────────────────────────────────────────┐
                  │             前端控制台 (React)           │
                  │  Sidebar 6 大分区 / RBAC 过滤            │
                  └───────────────┬─────────────────────────┘
                                  │ HTTPS + JWT
                  ┌───────────────▼─────────────────────────┐
                  │       后端 API (Gin)                    │
                  │  Middleware: Recover/CORS/RateLimit/JWT │
                  │  + Idempotency + RequireRole(admin)     │
                  └────┬─────────────┬───────────────┬──────┘
                       │             │               │
              ┌────────▼───┐  ┌──────▼─────┐  ┌─────▼──────┐
              │ PostgreSQL │  │   Redis    │  │  外部系统  │
              │ 业务+审计  │  │ 幂等+JWT   │  │ K8s/Helm   │
              │            │  │ 黑名单     │  │ Grafana    │
              └────────────┘  └────────────┘  │ N9E        │
                                              │ VLogs      │
                                              └────────────┘
```

控制流：
1. 前端发起 API（带 `Authorization: Bearer <jwt>` + 写操作时 `Idempotency-Key`）
2. 后端依次执行：限流 → 鉴权 → 黑名单校验 → 路由 → 处理器
3. 处理器调 service，service 调 repository / k8s / helm / grafana / n9e
4. 写操作落业务库 + 审计表，幂等结果写 Redis（TTL）

## 5. 多租户与权限

### 5.1 角色矩阵

| 接口类 | viewer | operator | admin |
|---|:-:|:-:|:-:|
| 查询（任意 GET） | ✓ | ✓ | ✓ |
| 接入安装/升级/卸载（本租户实例） | ✗ | ✓ | ✓ |
| 实例 scale（受策略约束） | ✗ | ✗ | ✓ |
| 模板/指标/日志实例/Grafana 主机/K8s 集群 CRUD | ✗ | ✗ | ✓ |
| 平台扩容 / 共享集群初始化 / 审计查询 | ✗ | ✗ | ✓ |

### 5.2 租户隔离

非 admin 用户的所有列表/详情接口在 service 层强制 `tenant_id == 当前用户租户`。

- 列表：`tenant_id` 过滤注入 SQL；分页统计同样过滤
- 详情：拿到对象后由 handler 调 `assertTenantAccess` 校验，跨租户返回 403
- 写操作（接入安装/卸载）：在 handler 入口和 service 内做双重校验

### 5.3 用户更新

- 普通用户只能改自己（邮箱、手机、密码）
- admin 可改任意用户，但角色变更只允许 admin 操作

## 6. 扩缩容策略（策略 A）

| 模式 | 实例水平 | 实例垂直 | 实例存储 | 平台级 |
|---|:-:|:-:|:-:|:-:|
| `shared` | ✗ | ✗ | ✗ | admin |
| `dedicated_cluster` | ✗ | ✗ | ✗ | admin |
| `dedicated_single` | ✗ | ✓ | ✓ | admin |

- 同一 `instance_id` 的并发 scale 由 `ScaleService` 内部互斥锁串行
- 每次 scale 落 `scale_events`：method ∈ `cr_patch` / `helm_upgrade` / `k8s_native` / `rejected`，无论成功失败
- 平台扩容仅作用于注册目标（`target_id`），不允许任意 namespace/name 输入

## 7. 接入中心生命周期

```
┌──────────┐  POST /integrations/templates           ┌─────────────────┐
│  admin   │ ───────────────────────────────────────▶│ Template (软删) │
└──────────┘  POST /integrations/templates/:id/versions
                    │                                  │ Versions
                    ▼                                  │ uk(tpl,version)
              version 快照 (collector/alert/dashboard) │
                                                       └────────┬────────┘
                                                                │
   user (本租户)  POST /integrations/install/plan  ─── dry-run ─┤
                  POST /integrations/install      ─── apply ────┤
                                                                ▼
   instance × template (partial uk where deleted_at IS NULL)  Installation
                                                                │
                                                                │ revisions
                                                                │ install / upgrade / reinstall / uninstall
                                                                ▼
   DELETE /integrations/installations/:id  →  软删 + 写 uninstall revision
   再次 POST /integrations/install 同 (instance,template)  →  复用旧 ID + action=reinstall
```

要点：

- 模板软删：`name` 是 partial unique（`where deleted_at IS NULL`），下架后可同名重建
- 版本删：`(template_id, version)` 唯一；版本被任何活跃安装引用 → 409；最后一个版本不允许删
- 安装活跃唯一：`(instance_id, template_id) where deleted_at IS NULL` 唯一
- 卸载是软删，**保留完整 revision 链**；`reinstall` 行为与 `install` 等价但更新原行
- 下发资源统一打 label：`managed-by=ops-system` / `template=<id>` / `installation=<id>`，Uninstall 时按 label 删

## 8. 数据存储

| 库 | 用途 | 关键表 |
|---|---|---|
| PostgreSQL | 业务 + 审计 | `users`/`tenants`/`instances`/`ops_integration_*`/`ops_metrics*`/`ops_log_instances`/`ops_grafana_hosts`/`clusters`/`scale_events`/`platform_scale_audits` |
| Redis | 幂等键 + JWT 黑名单 | key 形式：`idempotency:<key>`、`jwt:blacklist:<jti>` |

索引设计要点详见 `02-后端详细设计.md` §模型与索引。

## 9. 幂等与审计

- 写操作必须带 `Idempotency-Key`；首次请求 `SET NX + TTL`，重放命中则原样返回 + `X-Idempotency-Replayed: true`
- Redis 不可用时：写操作仍执行，但会在 audit 上打 `replayed=false`，并在日志里告警
- 审计三类（皆不软删）：
  - `platform_scale_audits`：平台扩容
  - `scale_events`：实例 scale
  - `ops_integration_installation_revisions`：接入变更

## 10. 错误处理

| HTTP | 场景 |
|---|---|
| 400 | 参数错误、UUID 解析失败、必填缺失 |
| 401 | 未登录、token 过期、token 在黑名单 |
| 403 | 角色不足、跨租户访问 |
| 404 | 资源不存在 |
| 409 | 重名、版本被引用、模板被活跃安装引用、最后一个版本删除 |
| 422 | 参数语义错误（如分页越界） |
| 429 | 限流 |
| 500 | 未分类异常（统一脱敏） |
| 503 | K8s/Helm/Grafana/N9E 等外部依赖不可用 |

前端 `extractApiError` 按 `后端 message → 状态码 → 网络错误 → fallback` 顺序产出文案。
