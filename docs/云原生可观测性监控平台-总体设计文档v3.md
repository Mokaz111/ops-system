# 云原生可观测性监控平台 - 总体设计文档（现网对齐版）

## 1. 文档目的

本文档描述 `ops-system` 当前已实现的控制面能力与系统边界，作为现网运行和后续迭代的基线文档。

约束：

- 若文档与代码冲突，以代码实现为准
- 本文档不描述已废弃的阶段性方案

## 2. 系统定位

`ops-system` 是可观测性平台控制面，负责：

- 组织与租户管理（部门、租户、用户、权限）
- 监控实例生命周期管理（VictoriaMetrics 侧）
- 平台级扩容与变更审计
- 与 Grafana、N9E 的管理接口集成

当前不包含：

- 自研告警引擎（告警引擎由 N9E 独立运行）
- 日志系统（VictoriaLogs）与链路追踪（Tracing）的完整产品化能力

## 3. 总体架构

核心链路：

1. 前端控制台（React + MUI）发起 API 请求
2. 后端网关（Gin）完成鉴权、限流、路由
3. 业务服务层调用数据库、K8s/Helm、Grafana、N9E 客户端
4. 关键管理操作（平台扩容）写入审计日志并支持幂等

## 4. 多租户与权限模型

- 角色：`admin` / `operator` / `viewer`
- 普通用户接口自动按租户/用户做数据隔离
- 管理接口（如实例创建、平台扩容）仅 `admin` 可访问

## 5. 扩缩容策略（策略 A）

- `shared`：实例级扩容禁止；仅平台级扩容
- `dedicated_cluster`：实例级扩容禁止；仅平台级扩容
- `dedicated_single`：允许实例级垂直扩容与存储扩容；不允许实例级水平扩容

平台级扩容对象采用注册目标清单（`target_id`），避免手填 namespace/name 误操作。

## 6. 平台级扩容与审计

扩容流程：

1. `dry_run=true` 生成变更计划（`spec_patch`）
2. 管理员确认后提交应用（`dry_run=false`）
3. 请求携带 `Idempotency-Key`，后端执行幂等保护
4. 无论成功/失败/重放，均落库审计

审计字段覆盖：

- 操作者（`user_id`/`username`/`role`）
- 客户端 IP
- 目标 `target_id`
- 状态（`success`/`failed`/`replayed`）
- 变更内容（`spec_patch`）
- 错误信息与时间

共享集群初始化流程（admin）：

1. 管理员调用初始化接口进行 dry-run 预览
2. 确认后执行 Helm `install or upgrade`
3. 固定 Chart 为 `vm/victoria-metrics-k8s-stack`
4. 初始化结果进入同一审计链路

## 7. API 分组（摘要）

- 认证：`/api/v1/auth/*`
- 部门：`/api/v1/departments/*`
- 租户：`/api/v1/tenants/*`
- 用户：`/api/v1/users/*`
- 实例：`/api/v1/instances/*`
- 告警集成：`/api/v1/alerts/*`
- Grafana：`/api/v1/grafana/orgs/*`
- 平台扩容（admin）：`/api/v1/platform/scaling/*`

## 8. 数据存储

- PostgreSQL：业务模型与审计日志
- Redis：平台扩容幂等键存储

## 9. 演进方向

- 在复用现有 K8s 资源封装能力基础上，逐步扩展到 VictoriaLogs、Tracing
- 完善平台审计查询体验与运维可观测能力
