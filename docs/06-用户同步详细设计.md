# 用户同步详细设计（现网对齐版）

## 1. 目标

确保平台用户、租户与外部系统（Grafana、N9E）的组织关系可关联、可追踪、可补偿。

## 2. 当前实现范围

平台内：

- 部门 / 租户 / 用户三级模型（`departments` / `tenants` / `users`）
- 角色：`admin` / `operator` / `viewer`
- 用户登录走 JWT；登出加入 Redis 黑名单（即时失效）

外部集成：

- Grafana：组织、组织用户、数据源（`/grafana/orgs/*`）
- Grafana 多主机：`/grafana/hosts`，按 `host_id` 解析独立 client
- N9E：用户/团队相关接口（封装在 `n9e.Client`）

约束：

- 平台元数据是主数据源
- 外部系统状态同步由平台触发，不假设外部主动回写

## 3. Bootstrap（首次部署）

- 接口：`POST /api/v1/users/bootstrap`
- 公共接口，**仅在系统中没有任何 admin 时允许**
- 已存在 admin 的二次调用 → `409`
- 创建动作落审计

## 4. 用户增删改

| 操作 | 谁可以做 |
|---|---|
| 创建（POST /users） | admin |
| 删除（DELETE /users/:id） | admin |
| 修改自己（邮箱 / 手机 / 密码） | 本人 |
| 修改任意用户 | admin |
| 修改角色 | 仅 admin（且不能把自己降级到唯一 admin 之外） |

密码哈希 bcrypt，cost 由 `OPS_BCRYPT_COST` 控制；存储字段不返回。

## 5. 同步触发点

| 平台事件 | 外部联动 |
|---|---|
| 创建租户 | 调 `grafana.CreateOrg` 创建对应组织，记录 `grafana_org_id` 到 `tenants` |
| 删除租户 | 调 `grafana.DeleteOrg`（最佳努力） |
| 创建用户 | 按角色与租户范围调 `grafana.AddOrgUser` |
| 删除用户 | 调 `grafana.RemoveOrgUser`（最佳努力） |
| 接入安装下发 dashboard | 选择 `grafana_host_id`，写入对应 host 的 org，回写 `installation.grafana_host_id` / `grafana_org_id` |

## 6. 一致性与失败处理

- 外部调用失败：记日志（不抛 5xx 阻断主流程，前端通过 `extractApiError` 透出后端 message）
- 关键标识（`grafana_org_id`、`grafana_host_id`）写入数据库后再对外返回成功
- 失败的外部同步可由 admin 通过对应管理接口重试 / 补偿（重新调用 grafana 接口）
- N9E 集成接口为代理调用，不在 DB 中冗余存储 N9E 实体

## 7. 租户隔离细节

- 普通用户（operator/viewer）的 `/users` 列表与 `/users/:id` 详情仅返回**同租户**用户
- `/tenants` 列表对普通用户仅返回自己所在租户
- `/instances` / `/integrations/installations` 等业务接口同样按 `tenant_id` 过滤
- 跨租户访问尝试统一返回 `403 ErrIntegrationTenantMismatch`（或同语义错误）

## 8. 安全要求

- JWT 密钥从 `OPS_JWT_SECRET` 注入，长度 ≥ 32
- 登出立即把 jti 写入 `jwt:blacklist:<jti>` Redis key，TTL = 剩余有效期
- Grafana / N9E 凭据从配置注入，不入代码仓库
- 日志不打印密码、JWT、API Token 任何字段
- 用户角色 / 租户绑定变更立即影响后续请求（无需重启），新 JWT 在下次签发时反映

## 9. 演进方向

- 增加密码策略（强度、历史复用、过期）
- 接入企业 SSO（OIDC / SAML），将 JWT 改为单点登录联邦校验
- 将 Grafana / N9E 同步动作纳入异步任务队列，提供"重新同步"接口给 admin 主动触发
