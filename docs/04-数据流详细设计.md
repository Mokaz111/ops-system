# 数据流详细设计（现网对齐版）

## 1. 控制面数据流

核心流程：

1. 前端发起管理请求
2. 后端完成鉴权、参数校验、权限校验
3. service 执行业务逻辑
4. repository 持久化业务数据
5. 必要时调用 K8s/Helm/Grafana/N9E 客户端

## 2. 平台扩容数据流

### 2.1 Dry-run

1. 管理员提交扩容参数（`dry_run=true`）
2. 后端校验目标与参数
3. 生成 `spec_patch` 计划并返回
4. 不写入 K8s 变更

### 2.2 Apply

1. 管理员提交扩容参数（`dry_run=false` + `Idempotency-Key`）
2. 后端先查幂等存储
3. 未命中则执行 VMCluster CR patch
4. 将结果写入幂等存储（TTL）
5. 写入审计日志

## 3. 审计查询数据流

1. admin 调用审计查询接口
2. 后端读取筛选参数（目标、状态、操作者、时间范围、分页）
3. repository 构建 SQL 条件并分页查询
4. 返回标准分页结构

## 4. 数据一致性原则

- 关键操作优先保证“可追踪”：先业务执行，再落审计（失败也记）
- 幂等重放请求不重复执行 K8s 变更
- 前后端字段命名保持一致，避免转换歧义
