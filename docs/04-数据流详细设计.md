# 数据流详细设计（现网对齐版）

## 1. 控制面通用数据流

```
前端 ──HTTP──▶ middleware（限流/CORS/JWT/Idempotency） ──▶ handler ──▶ service ──▶ repository ──▶ DB
                                                                       │
                                                                       └──▶ k8s/helm/grafana/n9e/vm/notify
```

每条业务请求都经过：

1. 限流 → 认证 → 黑名单 → 角色断言（必要时）
2. handler 完成参数绑定 + 租户/资源所有权断言
3. service 内编排业务规则、外部调用、审计写入
4. 写操作命中幂等存储 → 直接返回缓存结果

## 2. 平台扩容数据流

### 2.1 dry-run

```
admin POST /platform/scaling/vmcluster {target_id, spec, dry_run:true}
          │
          ▼
PlatformService.Plan
   1. 校验 target_id ∈ 注册目标
   2. 渲染 spec_patch
   3. 返回 patch
          │
          ▼
audits ← 写一条 status=success, replayed=false, spec_patch=...
```

### 2.2 apply

```
admin POST /platform/scaling/vmcluster {target_id, spec, dry_run:false} + Idempotency-Key
          │
          ▼
Idempotency.SetNX
   命中 → 直接返回缓存 + X-Idempotency-Replayed:true，audits.replayed=true
   未命中 ▼
PlatformService.Apply
   K8s MergePatch VMCluster.spec
          │
          ▼
audits ← status=success/failed, spec_patch=..., error_message=...
Idempotency.Set 缓存响应（成功/失败都缓存，TTL 24h）
```

## 3. 接入中心数据流

### 3.1 Plan（dry-run）

```
POST /integrations/install/plan {tenant_id, instance_id, template_id, version, parts, variables, cluster_id?, grafana_host_id?}
          │
          ▼
handler.assertTenantAccess(body.tenant_id)
          │
          ▼
service.Plan
   1. 加载 template + version 快照
   2. renderer 渲染 collector / vmrule / dashboard
   3. applier.DryRun → 计算 diff（与已存在资源比较）
   4. 返回 plan{rendered, diff}
（不落库）
```

### 3.2 Install（首次 / 重装）

```
POST /integrations/install + Idempotency-Key
          │
          ▼
查 ops_integration_installations where (instance_id, template_id) and deleted_at IS NULL
   存在 → 走升级路径（更新 version/parts/variables）
   不存在 → 检查软删历史：
        ├─ 有软删行 → 复用 ID，清 deleted_at，action=reinstall
        └─ 无       → 新建行，action=install
          │
          ▼
applier.Apply（按 cluster_id / grafana_host_id 路由）
          │
          ▼
revision ← {action, spec_diff, applied_resources, status, error_message}
installation.status = success / partial / failed
installation.last_revision_id = revision.id
```

要点：
- 资源 label 统一 `managed-by=ops-system` / `template=<id>` / `installation=<id>`
- `partial` 表示部分子下发器失败，错误细节在 revision.error_message 中
- failed 不会回滚已成功的子下发器（用户可重试或卸载后再装）

### 3.3 Uninstall

```
DELETE /integrations/installations/:id
          │
          ▼
service.Uninstall
   1. 按 label 收集已下发资源（k8s + grafana）
   2. applier.Delete（最佳努力，失败标记 status=uninstall_failed）
   3. installation.deleted_at = now、status=uninstalled / uninstall_failed
   4. revision ← action=uninstall, applied_resources=被删资源清单, status=success/failed
```

### 3.4 模板版本删除

```
DELETE /integrations/templates/:id/versions/:version
   1. 该版本被任意活跃 installation 引用 → 409 ErrIntegrationVersionInUse
   2. 该模板下只剩这 1 个版本 → 409 ErrIntegrationVersionLastOne
   3. 否则物理删除
```

### 3.5 模板删除

```
DELETE /integrations/templates/:id
   1. 该模板有任何活跃 installation → 409 ErrIntegrationTemplateInUse
   2. 否则软删 template + 软删其全部 versions
   （软删后 name 释放，可同名重建）
```

## 4. 实例 Scale 数据流

```
admin POST /instances/:id/scale {scale_type, replicas|cpu|memory|storage} + Idempotency-Key
          │
          ▼
ScaleService.Scale
   1. 互斥：sync.Map.LoadOrStore(instance_id, *Mutex)，Lock
   2. 加载 instance + tenant
   3. 策略 A 校验：
        - shared / dedicated_cluster → 直接 reject（method=rejected）
        - dedicated_single 且 horizontal → reject
   4. 选择 method：
        - VMCluster 类 → cr_patch（VMCluster.spec MergePatch）
        - Helm release → helm_upgrade
        - 普通 Deployment → k8s_native
   5. 执行 + 错误捕获
          │
          ▼
scale_events ← {scale_type, method, replicas/cpu/memory/storage, status, error_message, operator}
   3. Unlock
```

## 5. 指标库解析数据流

```
新增 / 更新 IntegrationTemplateVersion
          │
          ▼
metric_parser_service.Parse
   1. 解析 collector spec → 抽取 metric name + label 集
   2. 解析 dashboard JSON → 抽取 panel.expr 中引用的 metric
   3. 解析 alert spec → 抽取 alert 引用的 metric
   4. 写入 ops_metrics（按 (component, name) upsert，manual_override=true 跳过覆盖）
   5. 写入 ops_metric_template_mappings（区分 appears_in_collector/dashboard/alert）

POST /metrics/reparse/:templateId
   重新执行上述流程，仅对该模板下所有版本
```

## 6. 审计查询数据流

```
admin GET /platform/scaling/audits?target_id=&status=&operator=&start_time=&end_time=&page=&page_size=
          │
          ▼
repository.ListAudits
   按 created_at desc 排序，分页
          │
          ▼
返回 {items, total, page, page_size}
```

## 7. 一致性原则

- **可追溯优先**：业务操作执行后必须落审计（成功/失败/拒绝都写）
- **幂等不重放外部副作用**：Redis 命中即返回缓存响应，不再触发 K8s/Helm
- **租户隔离 in service**：避免遗漏，repository 不假设调用方做了过滤
- **字段一致**：前后端命名统一（snake_case → camelCase 由前端 type 显式映射），避免歧义
- **软删边界**：业务表用 `deleted_at`；审计表不软删，保证不可篡改
