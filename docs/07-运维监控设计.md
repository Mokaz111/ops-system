# 运维监控设计（现网对齐版）

## 1. 监控目标

- 平台 API 服务稳定性
- 数据库（PostgreSQL）与 Redis 可用性
- 外部依赖调用健康度（K8s / Helm / Grafana / N9E / VLogs）
- 平台扩容、实例 scale、接入变更三类审计链路完整可追溯

## 2. 基础健康检查

| 路径 | 含义 | 推荐告警 |
|---|---|---|
| `/health` | 进程健康 | 连续 3 次失败 |
| `/api/v1/health` | API 健康（含版本号） | 连续 3 次失败 |
| `/api/v1/health/db` | 数据库连通性 | 任意失败立即告警 |

## 3. 关键观测指标建议

| 维度 | 指标 |
|---|---|
| HTTP | 请求量、错误率（按 status code 聚合）、p50/p95/p99 |
| DB | 连接池占用率、慢查询数量、长事务数 |
| Redis | 连接可用率（影响幂等 + JWT 黑名单） |
| 平台扩容 | 成功率、失败率、replay 率、平均时延 |
| 实例 scale | 成功 / 失败 / rejected 计数（按 method 拆） |
| 接入安装 | install / upgrade / reinstall / uninstall 计数（按 status 拆） |
| 外部依赖 | grafana / n9e / k8s API 调用失败率 |

## 4. 审计与排障

平台维护三类审计表，全部不可软删：

| 表 | 触发场景 | 关键字段 |
|---|---|---|
| `platform_scale_audits` | 平台扩容 dry-run / apply / replay | 操作者、客户端 IP、目标、状态、`spec_patch`、错误信息、是否 replay |
| `scale_events` | 实例 scale（含 rejected） | scale_type、method（cr_patch/helm_upgrade/k8s_native/rejected）、replicas/cpu/memory/storage、operator、状态、错误 |
| `ops_integration_installation_revisions` | 接入安装 / 升级 / 重装 / 卸载 | action（install/upgrade/reinstall/uninstall）、spec_diff、applied_resources、status、error_message |

可用于：

- 变更追责（谁在什么时间做了什么操作）
- 故障窗口回溯（按时间窗口拉取所有变更）
- 依赖失败定位（applied_resources + error_message 联合分析）

排障路径示例：

- 实例突然不采集 → 查 `scale_events` + `ops_integration_installation_revisions`，筛选最近 1 小时该实例的所有变更
- 接入模板下发失败 → 看 `revisions.error_message` + applier 输出 + zap 日志中的 `integration_*` key
- 平台扩容看似没生效 → 查 `platform_scale_audits` 的 `replayed=true` 记录，确认是否被幂等拦截

## 5. 关键日志事件 key

按 zap key 检索：

- 鉴权 / 启动：`platform_idempotency_redis_unavailable`
- 集成下发：`integration_applier_enabled` / `integration_applier_noop` / `integration_seed_failed`
- 多集群：`cluster_inline_kubeconfig_ignored` / `cluster_k8s_client_init_failed`
- Helm / K8s：`scale_helm_client_init_failed` / `scale_k8s_client_init_failed`

## 6. 告警建议

| 告警项 | 触发条件 | 严重度 |
|---|---|---|
| DB 健康检查失败 | 连续 3 次 5xx | P1 |
| Redis 不可用 | 连续 5 分钟 | P2（功能降级，非阻断） |
| 平台扩容失败率 | 5 分钟内 > 30% | P1 |
| 实例 scale rejected 异常增多 | 5 分钟内 > 10 次 | P3（可能策略误用） |
| 接入安装 status=failed 占比 | 滚动 1 小时 > 20% | P2 |
| Grafana / N9E 调用失败率 | 5 分钟内 > 10% | P2 |
| JWT 黑名单写入失败 | 任何一次 | P2（涉及强制下线能力） |

## 7. 容量与备份

- PostgreSQL：业务表 + 审计表分库或分表归档（建议每月归档 `*_audits` / `scale_events` / `..._revisions`）
- Redis：保证幂等键 TTL 有上限，防止泄漏；JWT 黑名单 key TTL 自动清理
- 应用日志：建议落到本平台管理的 VictoriaLogs 实例，统一查询

## 8. 演进方向

- 暴露 `/metrics`（Prometheus / VM 兼容）端点输出关键计数（HTTP、scale、integration、外部调用）
- 审计冷归档与查询分层（热数据保留 90 天，冷数据走 OSS / 数据湖）
- 把 zap 关键事件转成结构化告警规则模板，纳入接入中心一并下发
