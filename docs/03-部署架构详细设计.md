# 部署架构详细设计（现网对齐版）

## 1. 组件关系

```
┌──────────────┐
│  前端静态站点  │  Nginx / 对象存储 + CDN
└──────┬───────┘
       │ HTTPS
┌──────▼─────────────┐    ┌──────────────┐
│  后端 API（Go）     │◀──▶│ PostgreSQL   │  业务 + 审计
│  Gin + GORM        │    └──────────────┘
│  Helm/k8s/Grafana  │    ┌──────────────┐
│  /N9E/VLogs 客户端 │◀──▶│   Redis      │  幂等 + JWT 黑名单 (可选)
└──┬──────┬──────┬───┘    └──────────────┘
   │      │      │
   │      │      └──▶ Kubernetes 集群（多个，按 ops_clusters 注册）
   │      │            ├─ VictoriaMetrics Operator (VMCluster/VMAgent/VMRule/VMPodScrape/VMServiceScrape)
   │      │            └─ VictoriaLogs (VLogs CR / Helm)
   │      │
   │      └──▶ Grafana 多主机（按 ops_grafana_hosts 注册）
   │
   └──▶ N9E（独立部署，告警计算 + 通知）
```

## 2. 后端部署要求

### 2.1 必备依赖
- PostgreSQL 13+：业务数据 + 审计
- Go 1.21+ 运行时
- 完整 `OPS_JWT_SECRET`（≥ 32 字符）

### 2.2 可选依赖
- Redis 6+：开启后才有完整幂等保护和 JWT 强制下线能力。Redis 不可用时：
  - 写操作仍执行
  - 幂等能力降级，日志告警 `platform_idempotency_redis_unavailable`
  - JWT 黑名单失效（依赖客户端自然过期）
- Kubernetes：通过 `kubernetes.in_cluster=true` 或 `kubernetes.kubeconfig=<path>` 注入凭据；不配置则相关接口降级为 `noop`（applier 自动切到 `NewNoopApplier`，仅返回 dry-run，不下发资源）
- Grafana / N9E：未启用时只影响对应业务模块，不阻断核心 API

### 2.3 环境变量
| 变量 | 用途 | 默认 |
|---|---|---|
| `OPS_JWT_SECRET` | JWT 密钥 | 无（必填） |
| `OPS_JWT_EXPIRE_HOURS` | JWT 有效期 | 24 |
| `OPS_BCRYPT_COST` | 密码哈希成本 | 10（生产建议 12+） |
| `OPS_DB_*` | DSN 各段 | — |
| `OPS_REDIS_HOST/PORT/PASSWORD/DB` | Redis 连接 | 关闭 |
| `OPS_KUBERNETES_IN_CLUSTER` | 集群内运行 | false |
| `OPS_KUBERNETES_KUBECONFIG` | kubeconfig 路径 | — |
| `OPS_RATE_LIMIT_*` | 限流参数 | — |
| `OPS_CORS_ALLOWED_ORIGINS` | CORS 白名单 | — |

## 3. 多 K8s / 多 Grafana 注册

### 3.1 K8s 集群
- 表 `clusters`，admin 维护
- 接入安装时按 `cluster_id` 解析 `k8s.Client`：
  - `cluster.in_cluster=true` 或 `kubeconfig_path != ""` 才会创建独立 client
  - inline `kubeconfig` 字段当前**不会自动落盘**（日志告警 `cluster_inline_kubeconfig_ignored`），需配合 path
  - 失败回退默认 client
- 客户端缓存：`k8s.NewClusterClientCache()` 按 `cluster_id` + fingerprint（in_cluster|path|inline_hash）缓存，避免每次 Apply 重做 discovery；并发安全

### 3.2 Grafana 主机
- 表 `ops_grafana_hosts`，admin 维护，`scope ∈ platform|tenant`
- 接入安装时按 `grafana_host_id` 解析 client；nil 或 host 失效时回退平台默认
- 凭据 `admin_token_enc` 加密列存储

## 4. K8s 操作边界

后端通过封装层执行：

| 类型 | 资源 | 实现 |
|---|---|---|
| 监控 | VMCluster CR `spec` MergePatch | `vm.SyncService` / `platform_scale_service` |
| 监控 | VMPodScrape / VMServiceScrape / VMRule | `integration.applier` |
| 应用 | Deployment 副本调整 | `scale_service`（method=k8s_native） |
| 存储 | PVC 扩容 | `scale_service` |
| Helm | install/upgrade（共享栈、VLogs） | `helm.Client` |

平台级扩容只允许注册目标，禁止任意 namespace/name 输入。

## 5. 运行时健康检查

| 路径 | 含义 |
|---|---|
| `/health` | 进程健康 |
| `/api/v1/health` | API 健康（含版本号） |
| `/api/v1/health/db` | 数据库连通性 |

不暴露 Redis 健康专用接口（其状态在启动日志和审计中体现）。

## 6. 安全与凭据

- 数据库密码、Redis 密码、JWT 密钥、Grafana/N9E API Token 全部从环境变量注入，禁止落入代码仓库
- bcrypt cost 通过环境变量按部署环境上调（开发 10，生产 ≥ 12）
- admin 接口必须叠加企业级访问控制（IAM / IP 白名单 / mTLS 等）
- 后端日志开启采集，至少保留：扩容审计、scale_events、integration revisions、登录/登出事件
- Grafana 主机 token 在 DB 中以 `admin_token_enc` 加密列形式保存，日志中不打印

## 7. 上线动作

1. 数据库迁移：首次部署后端会执行 `AutoMigrate`，所有 `ops_*`、`scale_events`、`platform_scale_audits` 等表自动创建
2. Bootstrap：`POST /api/v1/users/bootstrap` 创建第一个 admin，后续接口将拒绝重复 bootstrap
3. 共享集群初始化（admin）：
   - `POST /api/v1/platform/scaling/bootstrap/shared/init`
   - Chart 固定 `vm/victoria-metrics-k8s-stack`，默认目标 `monitoring/vm-shared-stack`
   - 先 `dry_run=true` 预览，再带 `Idempotency-Key` 应用
4. 注册多 K8s / Grafana 主机（如需）
5. 内置接入模板会在后端启动时由 `IntegrationSeeder` 自动 seed（失败仅告警，不阻断启动）

## 8. 日志与可观测

- 后端日志走 zap，建议接入 VictoriaLogs（可同样通过本平台的日志实例统一管理）
- 关键事件 key（用于检索）：
  - `platform_idempotency_redis_unavailable`
  - `scale_helm_client_init_failed` / `scale_k8s_client_init_failed`
  - `integration_applier_enabled` / `integration_applier_noop`
  - `integration_seed_failed`
  - `cluster_inline_kubeconfig_ignored` / `cluster_k8s_client_init_failed`
