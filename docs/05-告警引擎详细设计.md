# 告警引擎详细设计（现网对齐版）

## 1. 当前定位

告警计算与通知投递由 **N9E 独立部署**，`ops-system` 不实现自研告警引擎。

平台职责限定为：

- 通过接入中心模板的 `alert_spec` 声明告警规则
- 把规则下发到 K8s 的 VMRule CR（由 VMOperator 评估并产出告警事件）
- 提供 N9E 集成接口（规则、事件、通知渠道）以及告警查询/统计入口
- 在 InstanceDetail 提供"告警"Tab，展示该实例已下发的 VMRule 和 N9E 跳转入口

## 2. 告警下发链路

```
admin 在接入中心创建 / 升级模板版本
   alert_spec = {
     vmrule:  [...VMRule rule items...],
     n9e:     {...n9e 规则草稿...},          # 占位，当前 noop
     alert_targets: ["vmrule"]              # M2 只有 "vmrule"，未来可加 "n9e"
   }
          │
          ▼
用户在 instance × template 上 Install / Upgrade
          │
          ▼
integration.applier.vmRuleApplier
   按 cluster_id 解析 k8s.Client → 渲染 VMRule CR → Apply
   label: managed-by=ops-system / template=<id> / installation=<id>
          │
          ▼
VMOperator 监听 VMRule → 转换为 alertmanager rules
          │
          ▼
告警事件由 VictoriaMetrics + Alertmanager 流出（通常对接 N9E 或自有渠道）
```

`alert_targets` 是模板级别的开关，决定该模板版本要把规则推到哪些目标（`vmrule` / `n9e` / 未来其它）。当前仅 `vmrule` 落地，`n9e` 占位。

## 3. N9E 集成接口

平台提供以下能力（封装在 `n9e.Client` + `service/alert_*`）：

| 路径 | 用途 |
|---|---|
| `GET  /alerts/rules` | 查询 N9E 中的告警规则 |
| `POST /alerts/rules` (admin) | 创建 |
| `PUT  /alerts/rules/:id` (admin) | 更新 |
| `DELETE /alerts/rules/:id` (admin) | 删除 |
| `GET  /alerts/events` | 事件列表 |
| `GET  /alerts/events/:id` | 事件详情 |
| `PUT  /alerts/events/:id/ack` | 确认事件 |
| `GET  /alerts/channels` | 通知渠道列表 |
| `POST/PUT/DELETE /alerts/channels[/:id]` (admin) | 通知渠道维护 |
| `GET  /alerts/stats/{summary|trend|by-level|by-rule}` | 统计聚合 |

平台只是 N9E 的代理 + 缓存层，不做 SLA 承诺。

## 4. 权限规则

| 操作 | viewer | operator | admin |
|---|:-:|:-:|:-:|
| 查询告警规则 / 事件 / 渠道 / 统计 | ✓ | ✓ | ✓ |
| 确认事件（PUT ack） | ✗ | ✓ | ✓ |
| 创建/更新/删除规则与渠道 | ✗ | ✗ | ✓ |
| 通过接入中心下发 VMRule | ✗ | ✓（本租户实例） | ✓ |

数据可见性受租户隔离规则约束（与其它业务模块一致）。

## 5. 告警 InstanceDetail Tab

InstanceDetail 第 4 个 Tab 展示：

- 该实例上由接入中心下发的所有 VMRule（按 label `installation=<id>` 检索）
- 下发状态（来自 revision.status / revision.error_message）
- N9E 关联事件入口（外链跳转，平台不内嵌渲染）

## 6. 演进方向

- N9E AlertApplier 落地：`alert_targets` 增加 `"n9e"`，由 service 在 Apply 阶段调 `n9e.Client.UpsertRule`
- 告警分组与抑制规则模板化（与 VMRule 一同纳入接入中心模板）
- 告警事件回写到平台：用于 InstanceDetail 直接显示当前 firing 数

## 7. 运维要求

- N9E 可用性由独立运维体系保障；平台层只监控 `n9e.Client` 调用失败率
- 接入中心 VMRule 下发失败的 revision 必须可见（前端 InstanceDetail 已用 `statusLabel` / `actionLabel` + Alert severity 展示）
- 文档与前端文案明确"告警引擎独立运行"边界，避免使用方误以为平台承诺告警 SLA
